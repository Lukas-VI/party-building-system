# 模块维护说明

## 1. 目的
本文件用于说明当前项目中哪些模块沿用了已有稳定框架，哪些部分是业务定制层，以及继续开发时应优先参考的文档，避免后续在多端、多流程、多模型并存时失控。

## 2. 当前模块边界

### 2.1 `server/`
- 类型：项目业务核心
- 作用：认证、权限、流程、移动端工作台、统计导出、材料上传、微信绑定
- 当前特点：
  - 已从演示版收口到 MySQL + JWT
  - 新增 `/api/mobile/*` 作为服务号网页 App 主接口
  - 25 步流程仍是主流程模型
  - 前 12 步是第一阶段 MVP 办理边界，13 步以后暂不进入待办和办理接口
  - 后端写接口必须同时校验登录、权限、数据范围和流程节点状态
- 优先参考文档：
  - [electronic-dossier.md](electronic-dossier.md)
  - [project-overview.md](project-overview.md)
  - [deployment.md](deployment.md)

说明：
- `electronic-dossier.md` 当前仅用于整理《发展党员全程记实表》的阶段结构和研发参考口径
- 该文档本轮不等同于已落地的数据模型或页面实现

### 2.2 `admin-web/`
- 类型：PC 后台
- 框架来源：基于现有 React + Vite + TDesign 项目继续演进
- 保留原因：
  - 台账、统计、导出、配置更适合桌面端
  - 不宜和手机端共用一套布局
- 当前边界：
  - 不承载服务号工作台主交互
  - 主要负责管理型操作

### 2.3 `admin-mobile/`
- 类型：服务号网页 App
- 框架来源：
  - 继续沿用当前稳定的 `Vue 3 + Vite + Vant + Vue Router`
  - 这是已有成熟移动 H5 生态，不再从零造轮子
- 当前边界：
  - 申请人、审核者、学院党委共享同一入口
  - 只承担移动端高频办理、消息提醒、资料/材料维护
  - 不承担完整台账、复杂统计、组织配置
- 维护要求：
  - 页面信息架构优先围绕“待办、消息、流程、档案”
  - 不再按旧“移动后台台账页”思路扩展

## 3. 核心文档与代码的对应关系
- [electronic-dossier.md](electronic-dossier.md)
  - 对应：《发展党员全程记实表》的结构整理、阶段字段分组和研发参考边界
  - 关联代码：当前主要对应 `server/src/workflow-config.js` 的后续扩展参考，不代表档案模块已落地

- [project-overview.md](project-overview.md)
  - 对应：当前总体架构与阶段目标
  - 关联代码：整体目录、部署边界、模块职责

- [deployment.md](deployment.md)
  - 对应：`/DJ_api/`、`/web-admin/desktop/`、`/wx-app/` 的部署与反代
  - 关联代码：
    - `scripts/serve-admin-frontends.mjs`

## 4. 当前建议的开发顺序
1. 结合会议纪要补齐“25 步 -> 具体填写/提交内容”清单
2. 再决定是否独立建设电子档案模型
3. 视确认结果扩展服务号网页 App 页面和服务端模型
4. 最后再细化通知、签署、导出样式

## 5. 注释与模块化要求
- 对于框架保留代码，应在入口文件标明：
  - 使用的框架
  - 当前承担的业务职责
  - 对应文档
- 对于流程和档案规则，应尽量集中在配置文件中，不要散落在页面组件内部
- 对于新增接口，应优先沿用现有通用能力，而不是复制旧接口再做一套

## 6. 当前不应再做的事
- 不再继续把手机端主线建立在小程序页面之上
- 不再恢复 `pages/`、`utils/`、`miniprogram_npm/` 或根目录小程序入口配置
- 不再把“流程、材料、电子档案”混为一个页面概念
- 不再让桌面后台与手机端共用同一套页面结构
