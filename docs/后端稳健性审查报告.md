# 后端稳健性审查报告

审查日期：2026-04-27

审查范围：

- 服务端入口与配置：`server/src/app.js`、`server/src/env.js`、`server/src/db.js`
- 共享业务上下文：`server/src/app-context.js`
- 路由：`server/src/routes/*.js`
- 数据库初始化：`server/deploy/mysql-init.sql`
- 依赖清单：`server/package.json`

## 总体结论

后端已经具备可演示的主链路，但距离“可放心让业务人员持续跑数据”还有明显稳健性缺口。当前主要风险不是 SQL 参数化本身，而是权限边界、附件访问、流程状态一致性、数据库约束和生产配置兜底不足。

建议先处理 P0：敏感附件公开访问、全量用户列表暴露、上传绕过材料规则、流程审核缺少事务和并发保护。它们会直接导致数据泄露、错误流程推进或线上脏数据。P1 再补注册身份核验、数据库外键/约束、生产配置 fail-fast、错误信息收敛和审计可追溯。

本次只做审查报告，不修改代码。

## P0：阻塞上线/验收的高风险问题

### 1. 敏感材料通过 `/uploads` 公开静态暴露

- 代码证据：`server/src/app.js:44` 使用 `app.use('/uploads', express.static(env.UPLOAD_DIR))`；`server/src/app-context.js:1386` 的 `fileUrl()` 直接生成公开 URL。
- 当前问题：入党申请书、政审材料、培养考察材料等一旦上传，就可以通过 URL 直接访问，不经过 `requireAuth()`、数据范围校验或材料所属流程校验。
- 风险影响：只要链接泄露，任何人都能下载敏感政治材料和个人资料。对于党建系统，这是严重隐私与合规风险。
- 建议整改：停止公开挂载上传目录；改为 `/api/files/:id/download` 之类的鉴权下载接口，按 `attachments.step_record_id -> workflow_instances.applicant_id` 做数据范围校验；必要时增加一次性下载 token 和访问日志。

### 2. `/api/users` 登录即可获取全系统用户列表

- 代码证据：`server/src/routes/applicant-routes.js:19` 只使用 `requireAuth()`；查询没有 `scopeClause()` 或权限判断。
- 当前问题：任意已登录用户，包括普通申请人，都可以拿到所有用户的 `id`、`username`、姓名、单位、支部。
- 风险影响：泄露全校人员台账；也为后续越权探测提供用户 ID 和组织结构数据。
- 建议整改：如果用于候选人选择，应拆成受权限保护的候选接口，并按角色、单位、支部、正式党员身份过滤；普通申请人不应访问全量用户列表。

### 3. PC 通用上传接口绕过材料规则

- 代码证据：`server/src/routes/workflow-routes.js:169` 的 `/api/files/upload` 只要求登录，未校验文件存在、文件类型、材料标签、流程步骤、操作者身份；对比 H5 上传在 `server/src/routes/mobile-routes.js:312` 后才做流程和材料校验。
- 当前问题：任何登录用户都能通过 PC 上传任意扩展名文件，拿到公开 `fileUrl`。该文件不绑定步骤，也不进入材料审计。
- 风险影响：可上传不合规文件甚至可公开分发无关文件；材料链路验收时也无法证明文件属于哪个步骤。
- 建议整改：废弃或收紧通用上传接口；统一走带 `workflowId + stepCode + materialTag` 的上传接口；上传前检查 MIME、扩展名、大小、步骤状态和 actor 权限；失败时删除临时文件。

### 4. 流程提交/审核不是事务，容易出现半更新状态

- 代码证据：移动端审核先 `UPDATE workflow_step_records`，再 `advanceAfterReview()`，再 `logAudit()`/通知，见 `server/src/routes/mobile-routes.js:209`、`server/src/routes/mobile-routes.js:229`；PC 同样见 `server/src/routes/workflow-routes.js:107`、`server/src/routes/workflow-routes.js:127`；`advanceAfterReview()` 又会更新申请人阶段、流程实例和后续节点，见 `server/src/app-context.js:942`。
- 当前问题：步骤状态、申请人阶段、后续节点解锁/锁定、审计日志、通知分多条 SQL 执行，没有事务。如果中途失败，会留下“步骤已通过但下一步没打开”“身份已推进但日志缺失”等脏状态。
- 风险影响：核心流程数据不可追溯，验收跑几轮后容易出现前端显示和数据库真实状态不一致。
- 建议整改：把提交、审核、阶段推进、下一步状态变更、审计日志放进同一个 `pool.getConnection()` 事务；失败统一 rollback；成功后再发送非关键通知。

### 5. 流程审核缺少并发与状态版本保护

- 代码证据：审核接口读取 workflow 后直接按 `step.id` 更新，`UPDATE workflow_step_records ... WHERE id = :id` 没带旧状态或版本条件，见 `server/src/routes/workflow-routes.js:107`、`server/src/routes/mobile-routes.js:209`。
- 当前问题：两个管理员同时打开同一待审节点，都可以提交审核；后提交者会覆盖前者的 `form_data_json`、`review_comment`、`confirmed_at`。
- 风险影响：审核结果被覆盖，流程推进可能重复执行，审计记录也难以解释谁的操作最终有效。
- 建议整改：增加 `version` 字段或在 SQL 中带 `AND status IN (...)` 与受影响行数检查；审核开始时可用事务 `SELECT ... FOR UPDATE` 锁定当前节点。

## P1：核心稳健性缺口

### 6. 注册未与“预置身份证号/学工号/姓名”做三要素核对

- 代码证据：注册只从 `users` 查 `username` 和 `name`，见 `server/src/routes/auth-routes.js:71`；`registration_requests` 保存 `id_no`，但 `users` 或预置人员表没有身份证字段，见 `server/deploy/mysql-init.sql:30`。
- 当前问题：只要知道某个预置学工号和姓名，就可以用任意合法成人身份证号提交注册。
- 风险影响：首次注册身份核验不满足需求里的“后台预置人员核对、姓名/身份证/学工号一致性”。
- 建议整改：预置人员表或 `users` 增加身份证号哈希/脱敏字段；注册时做姓名、学工号、身份证一致性校验；错误原因继续保持明确但不要泄露过多枚举信息。

### 7. 注册提交会先改密码，审核失败或重复提交下存在账号污染

- 代码证据：`/api/auth/register` 插入 `registration_requests` 后立即 `UPDATE users SET password_hash`，见 `server/src/routes/auth-routes.js:90`、`server/src/routes/auth-routes.js:101`。
- 当前问题：账号尚未审核通过，用户密码已经被改写。若申请最终被拒绝，预置账号仍被外部提交者设置过密码。
- 风险影响：预置账号生命周期不清晰；如果后续管理员改为激活该账号，可能沿用旧申请设置的密码。
- 建议整改：把待注册密码暂存在注册申请表中，审核通过时再写入 `users.password_hash`；拒绝时丢弃。

### 8. 数据库没有外键、检查约束和状态枚举约束

- 代码证据：`server/deploy/mysql-init.sql` 只定义主键和少量唯一索引，`rg` 未发现 `FOREIGN KEY`、`CHECK` 或 `ON DELETE`；例如 `attachments.step_record_id` 见 `server/deploy/mysql-init.sql:123`，`workflow_step_records.instance_id` 见 `server/deploy/mysql-init.sql:106`。
- 当前问题：可以产生孤儿附件、孤儿角色、孤儿流程步骤；状态字段也可写入任意字符串。
- 风险影响：一旦后台导入、脚本或并发操作出错，数据库无法兜底，后续统计/导出会逐渐不可信。
- 建议整改：补齐外键；对 `status`、`task_status`、`scope_level` 等加枚举约束或应用层统一校验；上线前写迁移脚本清理历史脏数据。

### 9. 生产配置缺少 fail-fast，默认密钥和 CORS 容易误上线

- 代码证据：`server/src/env.js:19` 默认 `JWT_SECRET = 'change-this-secret'`；`server/src/env.js:31` 在 `NODE_ENV === 'development'` 时允许所有 CORS；若生产未正确设置 `NODE_ENV`，会变成宽松模式。
- 当前问题：服务启动时不校验生产必需配置，配置缺失也能运行。
- 风险影响：弱 JWT 密钥会导致 token 可伪造；CORS 放开会扩大浏览器侧攻击面。
- 建议整改：`NODE_ENV=production` 时强制校验 `JWT_SECRET`、数据库密码、`PUBLIC_BASE_URL`、CORS 白名单、微信配置；发现默认值直接启动失败。

### 10. `multipleStatements: true` 没有必要，扩大 SQL 注入后果

- 代码证据：`server/src/db.js:17` 开启 `multipleStatements: true`。
- 当前问题：当前大多数查询使用 named placeholders，整体参数化是好的；但开启多语句会让任何未来拼接 SQL 的漏洞后果更严重。
- 风险影响：一旦某处引入字符串拼接，攻击者可能从单条查询扩展到多语句破坏。
- 建议整改：关闭 `multipleStatements`；初始化脚本可用独立 raw connection 或迁移工具执行。

### 11. 多数接口直接把底层错误信息返回给前端

- 代码证据：大量 catch 使用 `fail(res, 500, error.message)`，例如 `server/src/routes/auth-routes.js:45`、`server/src/routes/workflow-routes.js:132`、`server/src/routes/org-routes.js:91`。
- 当前问题：数据库错误、字段名、约束名、内部异常信息会直接暴露给客户端。
- 风险影响：泄露内部表结构和实现细节，也不利于前端稳定处理错误码。
- 建议整改：生产环境统一返回“服务器内部错误”；详细错误写服务端日志；业务错误继续使用 `error.status` 和明确中文原因。

### 12. 审核待办列表只按数据范围，不按审核角色过滤

- 代码证据：`/api/reviews/pending` 只使用 `requireAuth()` 和 `scopeClause()`，见 `server/src/routes/review-routes.js:18`、`server/src/routes/review-routes.js:37`。
- 当前问题：同一单位/支部内，只要能看该范围，就能看到所有 `reviewing` 节点，不判断该节点是否属于当前用户角色可审核。
- 风险影响：待办数字和列表容易出现“看得见但不能办”或“无关角色看到敏感审核节点”的情况。
- 建议整改：增加 `requirePermission('review_steps')`，并按 `responsible_roles_json`/`workflow-config` 与用户角色交集过滤。

### 13. 公共 bootstrap 暴露测试账号提示和默认密码提示

- 代码证据：`server/src/routes/public-routes.js:17` 公开返回 `buildPublicBootstrap()`；`server/src/app-context.js:108` 查询登录提示账号；`server/src/app-context.js:150` 返回 `TEST_DEFAULT_PASSWORD_HINT`。
- 当前问题：未登录用户可以拿到可用账号提示和默认密码提示。
- 风险影响：验收阶段方便，但如果带到生产，会显著降低攻击门槛。
- 建议整改：仅在明确的演示环境开启；生产返回空数组或移除该接口。

### 14. H5 上传校验失败后没有清理 multer 临时文件

- 代码证据：`server/src/routes/mobile-routes.js:312` 使用 `upload.single('file')` 后才进入 `validateUploadedFile()`；catch 中没有删除 `req.file.path`。
- 当前问题：类型不合法、材料标签不合法、权限不合法时，文件已经落盘但不会删除。
- 风险影响：磁盘堆积；如果 `/uploads` 仍公开，还可能出现“校验失败但文件可访问”的情况。
- 建议整改：上传失败统一 `fs.unlink(req.file.path)`；更好是先落到非公开临时目录，业务校验通过后再移动。

### 15. `resolveReviewOutcome()` 将不合格业务结果回写为 `pending`，语义不稳定

- 代码证据：`server/src/app-context.js:1000` 到 `server/src/app-context.js:1003` 把“暂不确定/不合格/不同意发展”解析为 `pending`；`nextTaskStatus()` 又把 `pending` 映射为 `open`，见 `server/src/app-context.js:931`。
- 当前问题：业务上“不合格/不同意”是阻断结果，但系统状态显示为待处理/可打开，后续可通过改字段再次批准。
- 风险影响：验收时容易出现“已经不通过但状态仍待处理”的解释成本；审计上也不清楚这是退回补充、暂缓还是终止。
- 建议整改：拆分 `rejected`、`blocked`、`deferred`、`terminated` 等状态；业务暂缓需要有专门状态和恢复规则。

## P2：性能、可维护性和冗余问题

### 16. `canAccessApplicant()` 每次加载当前范围全部申请人

- 代码证据：`server/src/app-context.js:409` 调用 `getApplicants(user, {})`，再 `rows.some()`。
- 当前问题：单个详情/流程访问也会加载范围内所有申请人。
- 风险影响：数据量上来后，普通详情接口会变慢；全校管理员尤其明显。
- 建议整改：改成单条 SQL `WHERE applicant_id = :target` 加 scope 条件；返回是否存在即可。

### 17. `listMobileTodos()` 对每个申请人逐个查询流程，存在 N+1

- 代码证据：`server/src/app-context.js:1100` 遍历申请人后逐个 `getWorkflowByApplicantId()`。
- 当前问题：组织员/支部书记打开 H5 工作台时，申请人数越多 SQL 次数越多。
- 风险影响：验收数据少时不明显，真实学院数据量上来后首页会慢。
- 建议整改：待办列表用一条 SQL 查 `workflow_step_records + definitions + users`，只取可见待办节点。

### 18. `LIMIT ${Number(limit)}` 是潜在危险写法

- 代码证据：`server/src/app-context.js:1142`、`server/src/app-context.js:1236` 直接插入 `Number(limit)`。
- 当前问题：当前调用多为内部固定值，但 helper 本身没有 `Number.isFinite()` 和上下限夹取。
- 风险影响：未来一旦把 limit 暴露给 query 参数，可能导致 SQL 错误或异常。
- 建议整改：统一 `const safeLimit = Math.min(Math.max(parseInt(limit, 10) || 20, 1), 100)`。

### 19. PC/H5 提交审核逻辑重复，后续容易分叉

- 代码证据：`server/src/routes/workflow-routes.js:47` 到 `server/src/routes/workflow-routes.js:132` 与 `server/src/routes/mobile-routes.js:136` 到 `server/src/routes/mobile-routes.js:241` 大量重复。
- 当前问题：两端都手写表单合并、状态更新、审核结果解析、推进后续节点。新增规则时容易只改一边。
- 风险影响：PC 和 H5 行为不一致，尤其在材料校验、通知和错误提示上。
- 建议整改：抽成 `submitWorkflowStep()`、`reviewWorkflowStep()` service，路由只负责参数和响应。

### 20. 通知回执表没有在创建通知时写入

- 代码证据：`server/src/app-context.js:1199` 的 `createNotification()` 只插入 `notifications`；`notification_receipts` 仅在 `markNotificationRead()` 更新，见 `server/src/app-context.js:1179`。
- 当前问题：没有初始 receipt，点击时 `UPDATE notification_receipts` 可能更新 0 行。
- 风险影响：后续要验收“服务号推送闭环/回执”时，回执数据不完整。
- 建议整改：创建通知时同步插入 receipt；服务号发送成功/失败也写 receipt 状态。

### 21. `npm audit` 当前无法得到依赖漏洞结论

- 证据：在 `server/` 执行 `npm audit --omit=dev`，npm registry 为 `npmmirror` 时返回 `/-/npm/v1/security/* not implemented yet`。
- 当前问题：依赖安全审计没有跑通，不能证明依赖无已知漏洞。
- 风险影响：上线前安全基线缺一环。尤其 `xlsx` 属于解析外部文件的依赖，应保持关注。
- 建议整改：临时切到官方 registry 或 CI 中使用支持 audit 的 registry；同时限制导入文件大小、行数和格式。

## 冗余/不合理设计点

- `server/src/app-context.js` 同时承载鉴权、资料、流程、通知、上传、导出 helper，文件已经超过合理边界。建议按 `auth-service`、`workflow-service`、`notification-service`、`file-service` 拆分。
- 数据库初始化和运行时 schema 补丁混在 `server/src/seed.js`，容易让生产启动默默改表。建议使用明确的 migration。
- `workflow_step_definitions` 中有数据库配置，`server/src/workflow-config.js` 又有代码配置，当前运行时偏向代码配置。双源配置容易让后台改配置后不生效，需要明确“代码为准”或“数据库为准”。
- `profile_json`、`form_data_json`、`detail_json` 大量使用自由 JSON，短期开发快，但缺 schema 校验，后续导出/统计/审计会难以稳定。

## 建议整改顺序

1. 先关闭公开附件路径，建立鉴权下载接口，并统一 PC/H5 上传规则。
2. 修 `/api/users`、`/api/reviews/pending` 的权限和数据范围，避免普通用户或无关角色看到不该看的数据。
3. 给流程提交/审核加事务、并发版本检查和状态机约束。
4. 注册链路补预置身份证核验，并把密码写入延后到审核通过。
5. 生产启动增加配置校验：JWT、CORS、上传目录、PUBLIC_BASE_URL、数据库连接。
6. 数据库补外键、状态枚举和迁移脚本。
7. 把 PC/H5 重复工作流逻辑抽成 service，降低后续 13-22 步扩展风险。
